### 2. Computing User's IDLE Time

**PRINCIPLE: Let's decouple the SPA's knowledge of user's activity (or idle time) from how and when we update the server DB.**

I think that currently we might be updating users table (possibly too often) in particular — `last_seen_at` field by every API endpoint? Please investigate. We shouldn't be unless another update on that user is going on.

We are going to have a dedicated new endpoint to update the freshness of each user that will work based on the following logic:

A) We are going to create a "listen for everything from the user" event handler, that's invoked whenever the user is active. Please reference ReactJS documentation to make sure this is not a performance hog and that it's quick.

1. Whenever the handler is invoked, first of all it immediately stops listening, and does the following —

2. We either fetch UserState object from the ReactState or creates a brand new one object, which for now only has one attribute — the `last_seen_at` timestamp with time zone, and then we save this object to the React State. If the user refreshes the page, we might loose this state data temporarily, but we will initialize our interaction listener on page load, and that will get us our callback. The reactState here can be attached to the Chat component.

3. Right after we are called, we saved the `last_seen_at` into the ReactState, and then we re-enable our listener for user interactions, but we do so with a 15 second delay. 

4. Each time user's interaction invokes this handler, it just updates the `last_seen_at` attribute of some hash, which lives in the ReactJS state for the Chat Component. 

5. The 15 second delay is to debounce this so it doesn't happen too often.

> [!NOTE]
>
> Now we have last user's action timestamp in ReactJS State and it's pretty accurate.

6. Now, we'll need a *separate component*, a self-scheduler, which runs once per minute and it inspects the ReactState of the Chat component, and if it finds the `last_seen_at` there, it then looks for another attribute in the same state object called `activity_synced_at` — also timestamp. If if find it, we check if `last_seen_at` is less than `activity_synced_at` we return and do nothing. 

7. If, however, either that second timestmap is not in the ReactState yet, or it's older than the `last_seen_at`, then the client asynchronously sends an update to the server endpoint `POST /users/me/seen` with the body of the POST being the `last_seen_at` timestamp with zone. This call should return 200, after which we update `activity_synced_at` to now() and save the Chat's React State.

8. Meanwhile, on the backend: the route `/users/me/seen` reads the timestamp from the body, performs an async PG update on that user row (we know which user from `get_current_user()`) — but only if the DB value is older than the incoming value. Once we updated that one field, we return 200.

9. We can call this component `IdleComponent` or `UserActivityTracker` whatever. It fires on schedule once a minute while the SPA is loaded, but it ONLY generates the POST to the backend if and only if there is `last_seen_at` timestamp in the state, and either there is no `activity_synced_at` timestamp there too yet, or it exists and is older than `last_seen_at`. In the case when `activity_synced_at`  is newer than `last_seen_at` no backend call is necessary and we leave both timestamps alone. This component is sleeping for the next 60 seconds.

10. FastAPI endpoint: the client will invoke `POST /users/me/seen` and receives a JSON object of the form:

```json
{ "last_seen_at": "timestamp with timezone" }
```

If the update or commit fails, log the error to the log, and return HTTP 500 with error description.

---

There is no need to update the user's `last_seen_at` elsewhere, except in the /login and /logout routes, which should set it to NOW() with timezone.

### 3. Chatting without the Game

I want to make it possible for people to open the game, choose the AI game, start playing it, then maybe they get bored and they want to play (or chat) with @kate. So the open the "Chat" tab of the game, which should be behind settings at all times, whether or not the user is playing AI or human. 

This is where the user can figure out who is online using /who command, and if they find someone they want to invite into the chat (and the game) they type `/invite @kate`.

If @kate  has blocked @bob, @bob gets an error message in the chat right after his invite. It should say "Sorry, we couldn't deliver your message" and @kate does not get anything.

If @kate did not block @bob, then @kate gets a modal on her screen above all other modals, that tells her:

---
[  CHAT & GAME INVITATION! ]

Hey @kate! Another user @bob would like to chat and play a game with you. 

[ conditional — if /invite @kate message ] actually had the message, it's displayed here like so:

Oh, and there is an attached message from @bob! `message` 😃

[ Accept ]   [ Decline ]
---

So @kate now has a choice. If she declines, @bob gets a regular chat message from @kate "Apologies, but I can't play or chat at the moment."

If @kate accepts, it's equivalent to @bob giving her a code, and her entering the code and pressing [`Join`]. She should be dropped into a new game with @bob and be able to chat with him.

> For now all chats are 1-1 and attached to the game. This may not stay like that forever.
